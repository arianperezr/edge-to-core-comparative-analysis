Graphs Overview
===============

This project produces three primary graphs from each benchmark run. Together, they summarize the three pillars from the project overview:

- **Capability** — how much CPU work each platform can do.
- **Efficiency** — how well each platform uses storage and memory before hitting the "memory wall".
- **Resilience** — how quickly each platform recovers from injected faults (MTTR).

All of the graphs are generated by the plotter script:

    python3 analysis/plotter.py -o <results_directory>

For a single run directory (for example: `results/x86_64_20260313_121530`), you will typically see three PNGs next to the raw result files.

1) Capability: CPU Throughput (`plot_capability.png`)
-----------------------------------------------------

- **What it represents**
  - Overall CPU "capability" or throughput for each architecture under stress.
  - It answers: *"As I increase the number of CPU threads, which system keeps scaling and which one flattens out?"*

- **Axes**
  - **X‑axis**: number of `stress-ng` CPU threads.
  - **Y‑axis**: bogo operations per second (bogo ops/s).

- **How to read it**
  - Each **line** is one architecture (e.g., Edge Pi 5, Commodity Laptop, IBM Power10).
  - **Higher lines** mean more work completed per second.
  - **Error bars** (vertical black lines with caps) show run‑to‑run variability (± standard deviation).
  - If a line **flattens early**, that architecture hit a scaling limit (e.g., memory or scheduling bottlenecks).
  - If a line for Power10 **keeps rising** where x86 or ARM flattens, that is the "tipping point" where enterprise hardware continues to scale.

2) Efficiency: Storage / Memory Wall (`plot_efficiency.png`)
------------------------------------------------------------

This figure has two vertically stacked panels that share the same X‑axis (block size).

- **What it represents**
  - How efficiently each architecture uses the storage/memory subsystem under `fio` write workloads.
  - It answers: *"At which block sizes do we saturate bandwidth, and how does that impact tail latency?"*

- **Top panel — Bandwidth**
  - **Title**: "Efficiency sweep (fio write) — Bandwidth".
  - **X‑axis**: block size used for writes (4k, 64k, 1M, 4M, etc.).
  - **Y‑axis**: write bandwidth in MiB/s.
  - Each **colored bar** is one architecture; **taller bars** mean higher sustained bandwidth.
  - **Black lines** across the tops of bars are error bars (± standard deviation across runs).
  - A platform that **reaches high bandwidth at small block sizes** is generally more efficient.

- **Bottom panel — p99 latency**
  - **Title**: "Efficiency sweep — p99 latency".
  - **X‑axis**: same block sizes as the bandwidth panel.
  - **Y‑axis**: 99th‑percentile write latency in microseconds.
  - Each bar is one architecture; **shorter bars are better** (lower tail latency).
  - Error bars again show run‑to‑run variability.
  - When comparing architectures, look for the combination of:
    - **High bandwidth** (top panel) and
    - **Low p99 latency** (bottom panel)
    at the **same block sizes**. That is where a platform is both fast and well‑behaved under load.

3) Resilience: MTTR Under Fault Injection (`plot_reliability.png`)
------------------------------------------------------------------

- **What it represents**
  - Reliability of each architecture under SIFI (Software‑Implemented Fault Injection).
  - It answers: *"When we deliberately crash things, how long does each system take to recover?"*

- **How the data is produced**
  - SIFI runs are triggered via the collection harness (e.g., `./collect_data.sh`) with `ENABLE_SIFI=true`.
  - The harness:
    - Forces a simulated crash inside the container.
    - Measures how long it takes the system or service to recover.
    - Records MTTR samples (in seconds) to `mttr_data.csv` in the same results directory.

- **Axes**
  - **X‑axis**: architectures that have MTTR data (one bar per platform).
  - **Y‑axis**: Mean Time To Recovery (MTTR) in seconds.

- **How to read it**
  - Each **bar** is the mean MTTR for that architecture.
  - **Error bars** show the standard deviation across multiple SIFI runs.
  - **Shorter bars** mean faster recovery from injected faults — i.e., better resilience.
  - Large error bars indicate **unstable recovery behavior** (sometimes fast, sometimes very slow), which is a red flag even if the mean looks good.

Troubleshooting Missing Graphs
------------------------------

If an expected PNG is missing for a given run directory:

- **For capability and efficiency graphs**
  - Make sure the directory contains either:
    - `processed_results.json`, or
    - one or more `perf_run_*.json` files.

- **For the reliability graph**
  - Confirm that `mttr_data.csv` exists and has at least one SIFI run recorded.

To aggregate and compare multiple architectures on the **same** set of graphs, point the plotter at the parent directory that holds multiple result subdirectories, for example:

    python3 analysis/plotter.py -o results

The plotter will automatically pick up any child directories that contain valid data and overlay them on the same figures for direct comparison.
