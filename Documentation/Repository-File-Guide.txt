================================================================================
REPOSITORY FILE GUIDE — WHAT EACH FILE DOES
Edge-to-Core Comparative Analysis
================================================================================

This repository is a Docker-based validation harness for comparing edge,
commodity, and enterprise platforms:
- Edge: Raspberry Pi 5 (arm64)
- Commodity: x86_64 laptop/workstation
- Enterprise: IBM Power10 (ppc64le)

It measures three pillars:
1) Capability (stress-ng CPU sweep)
2) Efficiency (fio sweep across block sizes and concurrency)
3) Reliability (SIFI + MTTR)

Enterprise lab baseline assumption for documentation and analysis:
- ppc64le guest under KVM (para-virtualized), SMT1 (1 thread/core), 8 sockets.
- SMT1 is used as a per-thread fairness baseline when comparing enterprise to commodity x86 runs.

--------------------------------------------------------------------------------
ROOT-LEVEL FILES
--------------------------------------------------------------------------------

README.md
  What: Project summary and quick-start workflow.
  How:  Start here for run/collect/plot commands.

docker-compose.yml
  What: Compose service for the lab container.
  How:  Builds testbenches/Dockerfile, mounts ./results to /app/results, and
        defaults ENABLE_SIFI=false.

run_lab.sh
  What: Convenience wrapper for a single harness run.
  How:  Executes "docker compose build lab" then "docker compose run --rm lab".

collect_data.sh
  What: Scenario-driven repeated two-phase collection script.
  How:  Runs one or more scenarios (default: baseline,sustained,io_concurrency).
        For each scenario:
        - Phase 1 runs clean iterations and saves perf_run_<scenario>_*.json
        - Phase 2 runs SIFI iterations and appends scenario-tagged MTTR rows to
          mttr_data.csv as "<scenario>,<seconds>"
        Iterations are controlled by ITERATIONS (default 10).

COMMANDS
  What: Local command log file used by the workspace.

--------------------------------------------------------------------------------
CORE (Python harness)
--------------------------------------------------------------------------------

core/main.py
  What: Main harness entrypoint executed in the container.
  How:  Discovers architecture metadata, loads benchmark profile from env vars,
        runs capability + efficiency sweeps, optionally simulates SIFI failure
        when ENABLE_SIFI=true, then writes /app/results/processed_results.json.
        Result rows include scenario/concurrency metadata for analysis.
        Current output schema does not include perf cache counters, thermal
        sensor series, or direct power/efficiency-per-watt telemetry.

core/discovery.py
  What: Architecture classification helper.
  How:  Maps x86_64, aarch64, ppc64le, s390x to metadata used in outputs.

core/fabfile.py
  What: Legacy Fabric-based orchestration tasks.
  How:  Still present, but not part of the primary documented workflow.

--------------------------------------------------------------------------------
ANALYSIS
--------------------------------------------------------------------------------

analysis/plotter.py
  What: Produces comparison plots and summary CSV tables from harness outputs.
  How:  Reads processed_results.json or perf_run_*.json and mttr_data.csv when
        available, aggregates by scenario + workload keys, then generates:
        - plot_capability.png
        - plot_efficiency.png
        - plot_reliability.png
        - summary_capability.csv
        - summary_efficiency.csv
        - summary_reliability.csv

analysis/run_plotter.sh
  What: Venv-safe wrapper for running the plotter.
  How:  Creates/uses .venv, installs requirements, and invokes plotter.py.

analysis/requirements.txt
  What: Analysis dependencies (currently matplotlib).

--------------------------------------------------------------------------------
TESTBENCHES
--------------------------------------------------------------------------------

testbenches/Dockerfile
  What: Multi-architecture lab image definition.
  How:  Installs Python, stress-ng, fio, and sysbench; runs core/main.py.

testbenches/memory_wall.fio
  What: Optional fio profile for manual testing.
  How:  Not currently called by core/main.py directly.

--------------------------------------------------------------------------------
DOCUMENTATION
--------------------------------------------------------------------------------

Documentation/Project Overview
  What: High-level mission, lab tiers, pillars, and workflow.

Documentation/Graphs Overview.txt
  What: How to interpret capability, efficiency, and reliability plots.

Documentation/Midterm Project.md
  What: Course deliverable prompt package for systems-engineering writeups.

Documentation/Graduate-Lab-Report-Template.md
  What: Report template for documenting experimental results.

Documentation/Verification_Gap_Analysis.md
  What: Risk-to-mitigation traceability for hypervisor jitter and MTTR clock-skew controls.

--------------------------------------------------------------------------------
RESULTS
--------------------------------------------------------------------------------

results/
  What: Generated benchmark artifacts.
  How:  Typical run folder is results/<arch>_<timestamp>/ with
        perf_run_<scenario>_<n>.json and optional mttr_data.csv. Plotter outputs
        can also be written under results/ or a custom -o directory.

--------------------------------------------------------------------------------
NOTES
--------------------------------------------------------------------------------

- Primary workflow does not require Fabric.
- Key collection env vars include: SCENARIOS, ITERATIONS, CPU_SWEEP_DURATION_S,
  CPU_SWEEP_THREADS, FIO_RUNTIME_S, FIO_BLOCK_SIZES, FIO_IODEPTHS, FIO_NUMJOBS,
  FIO_RW, FIO_SIZE, and ENABLE_SIFI.
- Two image tags may appear in scripts: edge-to-core/lab:latest (compose) and
  assurance-harness (collect script). Both come from testbenches/Dockerfile.
